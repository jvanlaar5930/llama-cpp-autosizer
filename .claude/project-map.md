# Project Map: llama-cpp-autosizer

Generated by: create-claude-repo-memory

## Purpose

Cross-platform (Windows/Linux) terminal app that automatically benchmarks and tunes `llama-server` (llama.cpp) settings for a local GGUF model — iterating parameter changes until performance converges.

## Tech Stack

- .NET 10 / C# 13, single executable project (`AssemblyName: llama-autosizer`)
- Spectre.Console 0.49 (+ Spectre.Console.Cli) — TUI: menus, live tables, progress display
- Microsoft.Extensions.DependencyInjection / Http / Logging / Hosting (v9)
- System.Text.Json for all serialization (sessions, config, profiles, llama-server API DTOs)
- System.Management (Windows-only, conditional package ref) + P/Invoke `GlobalMemoryStatusEx` for hardware detection
- No test project and no CI/CD config exist in the repo (`Needs verification` if one is added later).

## Entry Points

- `src/LlamaCppAutosizer/Program.cs` — builds the DI container, wires services/UI, runs `MainMenu.RunAsync()`. Ctrl+C triggers graceful cancellation; `LlamaServerService` is disposed on exit to ensure the spawned `llama-server` process is killed.
- Run: `dotnet run --project src/LlamaCppAutosizer`

## Major Domains

See `.claude/domains/` for details:

- **Optimization loop** (`OptimizerService`, `RecommendationService`, `BenchmarkService`, optional `CloudAdvisorService` — Claude CLI as recommender with local-LLM fallback) — see [`.claude/domains/optimizer.md`](domains/optimizer.md)
- **llama-server process/API control** (`LlamaServerService`) — spawns/kills the server subprocess, calls its HTTP API (`/completion`, `/v1/chat/completions`, `/health`)
- **Hardware detection** (`HardwareDetectionService`) — VRAM/RAM/CPU scan, Windows (`wmic`/`System.Management`/P-Invoke) and Linux (`/proc/*`) paths, plus `nvidia-smi`/`rocm-smi`
- **Named profiles & sessions** (`ProfileLibraryService`, `SessionPersistenceService`) — persistence of saved configs and run history to `profiles/` and `sessions/`
- **TurboQuant integration** (`TurboQuantService`) — optional fork of llama.cpp with extra KV cache quant types; quantizes models via subprocess
- **TUI** (`UI/*`) — `MainMenu` top-level nav, `ProfileMenu`, `AppSettingsMenu`, `SettingsEditor`, `BenchmarkDisplay`, `MenuHelper`

## Known Gotchas

- `LlamaServerService` self-heals unsupported start settings (e.g. quantized KV cache without flash attention, or mmap failures) by retrying once with the setting reverted — `server.LastEffectiveSettings`/`LastStartAdjustmentNote` reflect what actually started, which may differ from what was requested. Downstream code (e.g. `OptimizerService`) must use the effective settings, not the requested ones.
- `RecommendationService.ExplorationOrder` is the canonical list of parameters the optimizer walks; adding a new tunable parameter requires updating this array and the corresponding apply/switch logic.
